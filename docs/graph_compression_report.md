# DiskANN Graph Compression Methods — Evaluation Report

## Overview

This document evaluates compression methods for the DiskANN `InMemStaticGraphReformatStore` neighbor ID storage. The graph stores neighbor IDs as `uint32_t` in a flat CSR (Compressed Sparse Row) layout. For large indexes, the graph dominates memory usage.

## Method A: Delta Encoding + Varint

### Algorithm

1. **Sort** each node's neighbor list in ascending order.
2. **Delta encode**: Store the first neighbor as-is, then store the difference between consecutive sorted neighbors.
   - Example: neighbors `[100, 105, 110, 200]` → first=100, deltas=`[5, 5, 90]`
3. **Varint encode** each value (first value + deltas) using variable-length integer encoding:
   - Each byte uses 7 data bits + 1 continuation bit (MSB).
   - Values 0-127 → 1 byte, 128-16383 → 2 bytes, etc.
4. **On decode** (`get_neighbours()`): Read varints sequentially, compute prefix sum to reconstruct original IDs. Uses `thread_local` buffer.

### Why It Works

Sorted neighbor deltas are typically much smaller than absolute IDs. For 29.9M nodes with degree 60, average absolute ID ≈ 15M (needs 4 bytes), but average delta ≈ 500K (needs 2-3 varint bytes).

## Method B: Delta Encoding + Bit-Packing

### Algorithm

1. **Sort** each node's neighbor list in ascending order.
2. **Delta encode**: same as Method A.
3. **Find max delta** per node to determine the minimum bit-width needed.
4. **Pack** all deltas using fixed-width bit fields (per-node width).
5. **Per-node layout in compressed buffer**:
   ```
   [bits_per_delta: 1 byte] [first_value: 4 bytes] [packed deltas: ceil(degree × bits / 8) bytes]
   ```
6. **On decode**: Read `bits_per_delta`, read first value, then unpack fixed-width bit fields using shift+mask. Uses `thread_local` buffer.

### Why It Works

Fixed-width bit-packing avoids the per-value overhead of varint continuation bits. If all deltas in a node fit in 19 bits, varint uses 3 bytes (21 bits + 3 continuation bits = 24 bits), but bitpack uses exactly 19 bits per delta.

### Trade-off vs Varint

- **Better compression**: No wasted continuation bits. Slightly smaller than varint when deltas are uniform.
- **Worse compression**: When deltas are highly variable within a node, the max delta sets the width, wasting bits on small deltas. Varint adapts per-value.
- **Faster decode potential**: Fixed-width extraction is SIMD-friendly (can unpack 4-8 values at once with SSE/AVX). Our current implementation is scalar.

## Method C: BFS Node Reordering + Delta + Varint

### Algorithm

1. **BFS reorder**: Starting from the graph's entry point (start node), perform breadth-first traversal. Assign new sequential IDs in BFS visit order. Nodes close in graph structure get close IDs.
2. **Remap** all neighbor IDs using the old-to-new mapping.
3. **Delta + Varint encode** using remapped IDs (same as Method A).
4. **On decode** (`get_neighbours()`):
   - Map the queried original ID → reordered ID to find compressed data.
   - Decode varint deltas to get remapped neighbor IDs.
   - Map each remapped neighbor ID back → original ID (via `_new_to_old` table).
   - Return original IDs so the data store lookups work correctly.

### Why It Works

In the original ID assignment, graph-local neighbors can have very distant IDs, producing large deltas. BFS reordering groups structurally close nodes with nearby IDs. Since the graph has strong locality (neighbors of neighbors are often shared), BFS-reordered deltas are much smaller.

### Trade-off

- **Best compression** of the three methods.
- **Extra memory**: Requires two mapping arrays (`_old_to_new`, `_new_to_old`), each `num_nodes × 4 bytes` = 228 MB total.
- **Extra decode cost**: Each decoded neighbor ID requires an array lookup to convert back to original ID.
- **Long load time**: BFS over 29.9M nodes takes significant time (~60s). Could be optimized.

## Method D: StreamVByte SIMD Delta Encoding

### Algorithm

1. **Sort** each node's neighbor list in ascending order.
2. **Encode** using `streamvbyte_delta_encode()` from the [StreamVByte library](https://github.com/lemire/streamvbyte) (Apache 2.0 license).
   - StreamVByte separates control bytes from data bytes, enabling SSSE3 vectorized decode.
   - Control: 2 bits per integer (encoding 1/2/3/4 byte data length).
   - Data: packed variable-length values without per-byte continuation bits.
3. **On decode**: `streamvbyte_delta_decode()` — SIMD-accelerated, processes 4 values at a time via shuffle tables.

### Why It Works

StreamVByte's SIMD decode eliminates the branch-heavy scalar varint loop. The separated control/data stream allows the CPU to prefetch and vectorize. Decode speed is 4-8x faster than scalar varint.

### Trade-off vs Scalar Varint

- **Slightly worse compression** (74.7% vs 72.4%) because StreamVByte uses 2 bits per value for control (4 lengths: 1/2/3/4 bytes), while scalar varint uses 1 continuation bit per byte (more granular).
- **Much faster decode** — SIMD vectorized, dramatically reduces P99.9 tail latency.

## Method E: BFS Reorder + StreamVByte SIMD Delta

### Algorithm

Combines Method C (BFS node reordering) with Method D (StreamVByte SIMD encoding):

1. **BFS reorder** node IDs from the graph entry point (same as Method C).
2. **Remap** all neighbor IDs to the new BFS order.
3. **Sort** remapped neighbors and **StreamVByte delta encode**.
4. **On decode**: StreamVByte SIMD decode + map remapped IDs back to original IDs via `_new_to_old` lookup.

### Why It Works

BFS reordering makes graph-local neighbors have nearby IDs, producing smaller deltas. StreamVByte efficiently encodes these small deltas and decodes them with SIMD. The combination leverages both techniques.

## Key Findings

1. **Best compression**: Method C (Reorder+Varint) at 66.2%, saving 1,712 MB of raw graph data. However, the `_new_to_old` mapping table adds overhead.

2. **Best tail latency**: Method D (StreamVByte) dramatically reduces P99.9 latency (2,221 us vs 5,939 us at L=10) due to SIMD decode. This is the most production-relevant metric.

3. **Reordering overhead**: BFS reordering (Methods C, E) adds ~60s to load time and requires two 228 MB mapping arrays. The `_new_to_old` lookup in `get_neighbours()` adds measurable decode overhead that partially offsets SIMD gains.

4. **Recommended for production testing**:
   - **Method D (StreamVByte)**: Best trade-off of compression (-19%), fast SIMD decode, lowest tail latency, simplest implementation (no reorder).
   - **Method B (Delta+Bitpack)**: Best compression without reorder (-22%), simple implementation, moderate decode cost.
   - Test both on production hardware where the noisy local P99.9 numbers will be more stable.

5. **License compatibility**: StreamVByte is Apache 2.0 (compatible with MIT). MaskedVByte is Apache 2.0. LZ4 lib is BSD. TurboPFor is GPL v2 (incompatible). FastPFOR is Apache 2.0 but block-oriented (poor fit for small arrays).

## Method F: MaskedVByte BMI2 Delta Encoding

### Algorithm

1. **Sort** each node's neighbor list in ascending order.
2. **Encode** using `vbyte_encode_delta()` from the [MaskedVByte library](https://github.com/lemire/MaskedVByte) (Apache 2.0 license).
   - Uses standard varint encoding (same format as Method A).
   - Decode uses `masked_vbyte_decode_delta()` which leverages **BMI2 PDEP/PEXT** instructions for SIMD-accelerated varint parsing.
3. **On decode**: `masked_vbyte_decode_delta()` — uses masked byte extraction for fast varint decode.

### Why It Works

MaskedVByte uses the same compact varint format as scalar varint (Method A), so compression ratio is identical. The decode speedup comes from BMI2 bit manipulation instructions that extract variable-width fields without branching.

### Trade-off vs StreamVByte

- **Better compression** than StreamVByte (72.4% vs 74.7%) because it uses true varint format (1 continuation bit per byte) rather than StreamVByte's 2-bit-per-value control scheme.
- **Decode speed depends on CPU**: BMI2 is fast on Intel but historically slow on AMD Zen 1/2. Modern AMD (Zen 3+) has fast BMI2.

## Method G: LZ4 Per-Node Delta Compression (Upper Bound Experiment)

### Algorithm

1. **Sort** each node's neighbor list in ascending order.
2. **Delta encode** into uint32 array (same as other methods).
3. **LZ4 compress** the delta byte stream per node using `LZ4_compress_default()`.
4. **On decode**: `LZ4_decompress_safe()` + prefix sum to restore original IDs.

### Result: Negative — LZ4 Expands the Data

LZ4 **increased** the data size to 101% of original (5,121 MB vs 5,068 MB). This is because:
- Each node's neighbor list is only ~240 bytes (60 neighbors × 4 bytes).
- LZ4 has per-block overhead (headers, match tables) that exceeds savings at this small block size.
- The delta-encoded uint32 values don't have the byte-level repetition patterns that LZ4 exploits.

### Conclusion

General-purpose block compressors are **not suitable** for per-node graph compression. The data blocks are too small and lack the redundancy patterns these algorithms target. Specialized integer compression (varint, bit-packing) is the correct approach.

3. **MaskedVByte** achieves the same compression as scalar varint (72.4%) with faster decode via BMI2, but P99.9 latency was worse than StreamVByte on our test machine (Intel). StreamVByte's separated control/data stream may be more cache-friendly.

4. **LZ4 is unsuitable**: General-purpose compressors expand the data at per-node granularity (~240 bytes). The overhead/block-size ratio is too high.

5. **Reordering overhead**: BFS reordering (Methods C, E) adds ~60s to load time and 228 MB for mapping arrays. The `_new_to_old` lookup in `get_neighbours()` adds measurable decode overhead.

6. **Recommended for production testing**:
   - **Method D (StreamVByte)**: Best trade-off — 19% compression, lowest tail latency, simplest implementation.
   - **Method B (Delta+Bitpack)**: Best compression without reorder (-22%), moderate decode cost.
   - **Method A (Delta+Varint)**: Same compression as MaskedVByte, simple, no external dependencies.

7. **License compatibility**: StreamVByte (Apache 2.0), MaskedVByte (Apache 2.0), LZ4 lib (BSD) — all compatible with MIT. TurboPFor (GPL v2) — incompatible.
